Overview

STURNUS is a support engineering training platform. Each challenge is a fictional customer incident — complete with a support ticket, server logs, database collections, and screenshots. Trainees investigate the evidence, query the MongoDB terminal, and answer multiple-choice questions to earn points.

The platform is built with vanilla HTML/CSS/JS, hosted on Vercel, and backed by Supabase (Postgres + Auth + RLS). There is no build step — every file is served as-is.

Two platforms, one codebase. STURNUS (sturnus.app) and V.H.S. (bekind.support) share the same code and database but have separate themes, leaderboards, and user pools. The theme is determined by the domain at login and stored per-user. Admins can see both.

Admin Panel

The admin panel is at /admin.html. It is only accessible to users with role = 'admin' in the profiles table. The link is hidden from regular users.

Challenges tab

The main challenge management interface. From here you can:

  • Browse all challenges with search, status filter, and sortable columns
  • Toggle active/inactive per row, or use checkboxes for bulk activate/deactivate
  • Click ＋ New Challenge to open the creation form
  • Click ⬆ Import to bulk-import a JSON array of challenges — any challenge with id: "TICKET-XXXX" is automatically assigned the next available ID. IDs are still editable in the preview before confirming.
  • Use the ⋯ row menu to Preview, Edit, Duplicate, or Delete a challenge
  • The Difficulty column shows the community average rating from post-mortem reflections
Inactive challenges are hidden from users on the dashboard. Use this to stage a challenge before it's ready, or to retire one without deleting it.

Analytics tab

Shows per-challenge stats: attempt count, solve count, solve rate, average score, average hints cost, and average time to solve. Rows are flagged automatically:

  • Red border — solve rate above 90% (possibly too easy) or below 20% (possibly too hard). Only triggered with ≥3 attempts.
  • Yellow border — average hints cost exceeds 50% of average score (hint-dependent).

Use these signals to decide whether a challenge needs rebalancing — adjust question difficulty, add a free hint, or change point values.

Users tab

Shows all users with their handle, score, theme, and role. Use Reset scores to zero out a specific user's score and clear their submissions — useful during testing.

Invite tab

Sends a magic-link invite email via the /api/invite serverless function. The invitee clicks the link, picks a handle, and is in. You can invite as user or admin. Invites only work when deployed to Vercel (the function requires the service role key which is never in client code).

Writing Challenges

The best challenges have a clear root cause hidden behind plausible red herrings, at least one data source that requires querying, and questions that test reasoning not trivia. A good challenge takes 15–30 minutes to solve without hints.

Structure

Every challenge has:

  • ID — e.g. TICKET-0004. Auto-incremented by the form, but editable.
  • Title — short, evocative name for the incident (e.g. "The Phantom Session").
  • Scenario — 2–4 sentences describing the incident from a support/engineering perspective. What happened, when, how widespread.
  • Ticket quote — the customer's actual words. Ideally emotional and specific enough to feel real. This is what the trainee sees first.
  • Architectural context — the relevant infrastructure: services, databases, caching layers, deployment cadence. Trainees use this to orient their investigation.
  • Red herrings — a JSON array of plausible but irrelevant facts. These make the challenge feel realistic and harder.
  • Tags — used for filtering and "recommended next challenge" matching. Keep them lowercase and consistent (e.g. redis, session-management, incident-response).
  • Active — must be checked for the challenge to appear to users.

Questions

Each challenge has 2–4 questions that unlock progressively as the trainee answers correctly. Each question has:

  • Label — a short category shown above the question (e.g. "Root Cause", "Immediate Mitigation").
  • Question text — the actual question. Should be specific and unambiguous.
  • 4 options — one correct, three plausible distractors. Distractors should be wrong for a clear reason a knowledgeable person would understand.
  • Correct option — zero-indexed (0–3).
  • Points — typically 100–150 for root cause, 75–100 for follow-up questions.
  • Explanation — shown after submission. Explains why the correct answer is right and why the distractors are wrong. This is the most educational part.
Tip: write the explanation first. If you can't clearly articulate why one answer is right and others are wrong, the question isn't ready yet.

Hints

Hints cost points when unlocked. Convention:

  • Hint 0 (free) — a broad directional nudge. "Look at both log files — the timestamps are related."
  • Hint 1 (25 pts) — explains a key technical concept the trainee might not know.
  • Hint 2 (50 pts) — points directly at the relevant evidence.

Hints should never give away the answer outright — they should reduce the search space.

MongoDB collections

Each challenge includes synthetic MongoDB documents that trainees query via the in-browser terminal. The terminal supports find, findOne, countDocuments, and distinct with full filter, projection, sort, and limit support.

Good collection design:

  • Include 2–3 collections — enough to cross-reference, not so many it's overwhelming
  • Use realistic field names and data types ($oid, $date for ObjectIds and timestamps)
  • Plant the key evidence in the data, but don't make it the only document
  • Include a few normal-looking documents alongside the smoking gun

In the admin form, the MongoDB field is a JSON array. Each entry needs collection (name), required_field (used for schema display), and synthetic_docs (array of documents).

Log files

Log files are shown in a syntax-highlighted modal. Include 1–2 log files per challenge as JSON arrays of log entries. Each entry should have at minimum ts (ISO timestamp), level (info/warn/error), and msg. Add relevant fields (e.g. session_id, user_id, path) to make queries meaningful.

The most effective log files show a pattern over time — a warning escalating to errors, or a cascade starting with an unrelated-looking event.

Contributing

Local development

The app has no build step. To run locally:

  1. Clone the repo and open any HTML file directly in a browser, or use a local server (e.g. npx serve . or VS Code Live Server).
  2. Create a .env.local file (or set js/config.js directly) with your Supabase project URL and anon key. The anon key is safe to commit — RLS handles data isolation.
  3. Run the schema against your Supabase project's SQL editor: start with schema.sql, then apply any migrate-*.sql files in order.
  4. Seed a test challenge using seed-test-ticket.sql as a reference. Set active = false until it looks right.
Never commit the service role key. It lives only in Vercel environment variables and is only used by the /api/invite serverless function. The anon key in js/config.js is intentionally public.

Deploying

The app is deployed to Vercel from the main branch. Push to main and Vercel auto-deploys within ~30 seconds. No build command — Vercel serves the static files directly.

Vercel environment variables needed:

  • SUPABASE_URL — your project URL
  • SUPABASE_SERVICE_ROLE_KEY — only used by /api/invite.js

The vercel.json config handles routing. All .html files are served as-is with no rewriting.

Schema changes

Schema changes go in a new migrate-*.sql file. Run it in the Supabase SQL editor, commit the file alongside the front-end changes that use it. Naming convention: migrate-{feature}.sql (e.g. migrate-notes.sql).

A few things to know:

  • Always use CREATE TABLE IF NOT EXISTS and DROP POLICY IF EXISTS before CREATE POLICY so migrations are re-runnable.
  • If you need to change a Postgres function's return type, you must DROP FUNCTION first — CREATE OR REPLACE won't work for signature changes.
  • All user-facing tables need RLS enabled and appropriate policies. The pattern is auth.uid() = user_id.
  • RPCs that write data should be SECURITY DEFINER so they run with elevated privileges, but always validate auth.uid() inside the function body.
